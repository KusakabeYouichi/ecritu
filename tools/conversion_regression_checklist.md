# 変換回帰チェックリスト(LM再学習・大改修時の検証用)

このセッション(edition 1909〜1938)で個別対応した「報告読み」の期待結果。
LM(word_lm.json.gz)を再学習した後や連文節ロジックを触った後は、ここを一通り
確認して黙って壊れていないかを見る。多くは実機状態(抑制/追加語彙の播種)に依存する
ため、Macのエンジン直呼びだけでは再現しないものがある(下の「依存」を参照)。

検証方法:
- 実機: ⌘R でビルド→アプリを一度開いて播種→各読みを打って第1候補を確認。
- Mac(概算): DIAGテストで `candidates()` / `multiClauseCandidates()` を直呼び。
  抑制依存は `ÉcrituSuppr_Vocab` に InitialSuppr(+Hidden)を注入、
  追加語彙依存は `addUserEntry` で該当を注入して再現する(initialUserDictionaryは
  テストバンドルで空になるため)。

凡例 [依存]: suppr=抑制, curated=追加語彙(sacoche/misc), seed=KanaKanjiSeedDictionary,
rule=活用規則, code=連文節/ランキングのコード側ロジック。

## 実機劣化の根本(edition 1997)
連文節LM(400MB sqlite)はメモリ警告でアンロードしない(clearAllCacheはJSONのみ破棄・sqlite保持)。
アンロードすると hasWordLMMetadata=false で連文節が単文節に劣化し「実機だけ者芯」型の
断片変換が断続発生する。simでは再現せず。実LM再現は Tests でコンテナへ tmp sqlite をコピー。

## curated全長語+助詞の崩れ(既知の構造的弱点)
のみにいけます(curated完全一致)は全長1ノード→連文節が return [](pathIndices<2)で
単文節に任せ、追加語彙2400がクリーンに1位。だが のみにいけますよ(+助詞)は curated が
部分ノード[0-7]+よ で2ノード化→連文節ラティスが再開し 耳/生ける 等の競合が表面化する。
+1文字で「単文節curated経路→連文節ラティス」に切替わるのが境界(line ~463)。頻発するなら
「curatedノードを含む最良経路はその区間を確定扱いにして変種競合させない」構造対応を検討。

## 追加語彙の連文節参加(edition 2001)
vin/it/ryukyu 等の独自語(Sudachiコスト無し)は build_kana_kanji_sqlite で既定
word_cost を付与=連文節ラティスに参加。読み4文字以上=7500/3文字以下=11000
(短い読みのレア固有名 1ノード hijack 防止)。Sudachi実在語のコストは上書きしない。
⌘Rで辞書再生成後に有効。4文字以上でレア固有名が一般句を潰す再発が出たら 7500 を上げる。
